浏览器加载机制

HTTP协议

HTTP协议基础知识

加载，即为获取资源文件的过程，使用timeline来做测试

让外部引用文件（css、js、images）将他们放到html合适的位置，使浏览器以最快的速度将文件加载完毕。

prefetch 优化

提前下载js、css文件，预加载js并不会改变dom结构，他将这个工作留给主加载。

JS优化方案：

现象：JS会阻塞后续资源下载

原因：JS有可能会修改DOM，最为经典的document.write，这意味着，在JS执行完成前，后续所有资源的下载可能是没有必要的，这是js阻塞后续资源下载的根本原因。
办法：可以将外部引用的js文件放在底部的</body前。

现象：css文件不影响js文件的加载，但影响js文件的执行

原因：当js文件中有操作类似 var width = $(‘#id’).width()，这意味着，js代码执行前，浏览器必须保证css文件已下载和解析完成，才能再进行width获取。这也是css阻塞后续js的根本原因。
办法：当js文件不需要依赖css文件时，可以将js文件放在头部css的前面。

现象：调用js文件运行时间较长的函数

原因：浏览器常驻以下五个常驻线程：
1.浏览器GUI渲染线程;
2.javascript引擎线程;
3.浏览器定时器触发线程（setTimeout）;
4.浏览器事件触发线程;
5.浏览器http异步请求线程（.jpg 这类请求）
办法：使用setTimeout函数调用3线程。

解析，即把html文档解析成解析树

在构建DOM结构，组织css选择器时，选择最优的写法，提高浏览器的解析速率。

  • css解析将css文件解析为样式表对象。#test p{ color:#999999}遍历是自右向左，也就是先查询到p元素，再找到上一级id为test的元素。解决办法：共享样式数据。
  • js解析因为文件在加载的同时也进行解析

渲染，即为构建渲染树的过程，他是原来DOM树的可视化表示

在设置元素属性，编写js文件时，可以减少”重绘“”重新布局“的消耗。

  • 渲染最大的一个困难就是为每一个dom节点计算符合他的最终样式。
  • display:none不会被插入渲染树中
  • 节点的位置为绝对或浮动定位，脱离文档流，渲染树标识出真实的位置。

webkit渲染

WebKit 是一个开源的浏览器引擎，WebKit的优势在于高效稳定，兼容性好，且源码结构清晰，易于维护。

Gecko渲染

Gecko是套开放原始码的、以C++编写的网页排版引擎。目前为Mozilla家族网页浏览器以及Netscape 6以后版本浏览器所使用。
Mozilla开发者将在2017年逐步用Servo引擎替代古老的Gecko渲染引擎

基础优化

html规范:

  • dom结构不要太深
  • 符合W3C规范，很多做不到完全一致W3C统一验证工具
  • 注释要有

css规范:

  • 避免使用通配符
  • 不要为id选择器指定类名或是标签，因为id可以唯一确定一个元素
  • 多用class类，增强可复制性
  • 处理清空默认样式外，不要直接在标签上加样式。
  • margin、padding等可以简写的不使用简写，便于维护（腾讯和阿里推荐）
  • 减少内联样式
  • 避免后代选择器#tp p{} ，使用子选择器#tp>p{}
  • 注释

images:

  • 减少图片加载，使用雪碧图sprites合并
  • 压缩图片的大小

JS规范：

  • 减少全局变量
  • 能够减少代码情况下进行代码精简
  • 高内聚，低耦合(一个函数只做一件事)
  • 减少dom操作(少用类似getElementByTagName()这类耗用操作的方法)
  • script统一放在html底部

中级优化

  • 减少重定向（引用其他站点的图片、文件）
    status：200第一次加载，304缓存
  • 资源懒加载，减轻服务器压力，加快网页速度（代码渲染速度）。
  • 图片压缩
  • css、js文件合并，减少请求
  • 改小图片为图标字体（iconfont，可以兼容到ie7）

高级优化

  • 代码选择调用（组件化开发、重用性强）
  • 将不是第一次需要渲染的代码封装起来
  • 异步加载资源（图片预加载+选择加载）
  • 使用CDN，帮助服务器尽可能的避开互联网上影响数据传输和稳定性的环节（网络拥堵）。使内容加载更快、更稳定。让用户就近选取服务节点获取资源。
  • 功能模块化（requireJS、SeaJS），达到代码高内聚、低耦合。便于协调合作开发。兼容ie6+以上（重点）
  • gulp工程化工具进行优化（重点）
