Git概述

廖雪峰git实践

什么是Git？

Git是一款开源的分布式版本控制系统。主要针对于文本文件（TXT文件，网页，所有的程序代码等非二进制文件[音频、视频、word文档等等]）的修改管理。

为什么要用Git？

  • 功能强大
  • 不需要联网
  • 安全性要高
  • 开源而且免费

Git原理解析

基本概念

  • 工作区（Working Directory）-本地电脑文件目录
  • 版本库（Repository）- 工作区内的隐藏目录.git
  • 暂存区（stage或者叫index）：保存提交已经被修改的文本。
  • 分支 （主分支master）：自动创建的第一个分支master
  • 指针 （head）：默认指向主分支master
  • 远程仓库：github远程服务/自建git服务

原理

Git管理的是修改，而不是文件。
什么是修改？比如你新增了一行，这就是一个修改，删除了一行，也是一个修改，更改了某些字符，也是一个修改，删了一些又加了一些，也是一个修改，甚至创建一个新文件，也算一个修改。


工作区修改的文件,只有被git add <files>暂存区后修改，才能被git跟踪管理。

建议：git commit前后，我们再运行git status看看当前仓库的状态；
常见状态有：

  • 新建文件status——未跟踪（Untracked）
  • add暂存区内status - 改变（changes 撤销rm、提交选择commit）
  • 工作区中删除已经add到暂存区的文件status - change
  • 工作区里的文件已经add暂存区，之后又再次进行修改后的status
  • commit暂存区status - 没有变化（nothing工作区与暂存区没有需要提交的改变）
  • 编辑工作区内的文件status -改变（changes，提交暂存区或丢弃工作区的修改）

  • add工作区里已经改变的文件到暂存区status - 改变（changes，撤销暂存区的改动）

  • commit暂存区里的修改status - 没有变化（nothing）

版本回退

每当你觉得文件修改到一定程度的时候，Git就可以“保存一个快照”，这个快照在Git中被称为commit。一旦你把文件改乱了，或者误删了文件，还可以从最近的一个commit恢复，然后继续工作，而不是把几个月的工作成果全部丢失。

标签管理

发布一个版本时，我们通常先在版本库中打一个标签（tag），这样，就唯一确定了打标签时刻的版本。将来无论什么时候，取某个标签的版本，就是把那个打标签的时刻的历史版本取出来。所以，标签也是版本库的一个快照。

创建标签

$ git branch
$ git checkout master

打标签

$ git tag //默认为HEAD
$ git tag //对特定版本打标签
$ git tag -a -m “version 0.1 released” //用-a指定标签名，-m指定说明文字
$ git tag -s -m “signed version 0.2 released”可以用PGP签名标签

查看标签

$ git tag

查看标签信息

$ git show //$ git show v0.9

操作标签

  • $ git push origin <tagname>可以推送一个本地标签；
  • $ git push origin --tags可以推送全部未推送过的本地标签；
  • $ git tag -d <tagname>可以删除一个本地标签；
  • $ git push origin :refs/tags/<tagname>可以删除一个远程标签。

远程仓库

Git是分布式版本控制系统，同一个Git仓库，可以分布到不同的机器上。怎么分布呢？最早，肯定只有一台机器有一个原始版本库，此后，别的机器可以“克隆”这个原始版本库，而且每台机器的版本库其实都是一样的，并没有主次之分。

github注册远程仓库

创建SSH Key

$ ssh-keygen -t rsa -C “youremail@example.com”

计算机用户主目录下，自动生成一个.ssh目录，里面有id_rsa和id_rsa.pub两个文件，id_rsa是私钥，不能泄露出去，id_rsa.pub是公钥，可以放心地告诉任何人。

登陆GitHub，打开Account settings-SSH Keys-Add SSH Key，填上任意Title，在Key文本框里粘贴id_rsa.pub文件的内容。

GitHub允许你添加多个Key。假定你有若干电脑，你一会儿在公司提交，一会儿在家里提交，只要把每台电脑的Key都添加到GitHub，就可以在每台电脑上往GitHub推送。

工作建议

分支策略

在实际开发中，我们应该按照几个基本原则进行分支管理：
首先，master分支应该是非常稳定的，也就是仅用来发布新版本，平时不能在上面干活；
那在哪干活呢？干活都在dev分支上，也就是说，dev分支是不稳定的，到某个时候，比如1.0版本发布时，再把dev分支合并到master上，在master分支发布1.0版本；
你和你的小伙伴们每个人都在dev分支上干活，每个人都有自己的分支，时不时地往dev分支上合并就可以了。
所以，团队合作的分支看起来就像这样：

远程管理：

  • master分支是主分支，因此要时刻与远程同步；
  • dev分支是开发分支，团队所有成员都需要在上面工作，所以也需要与远程同步；
  • bug分支只用于在本地修复bug，就没必要推到远程了，除非老板要看看你每周到底修复了几个bug；
  • feature分支是否推到远程，取决于你是否和你的小伙伴合作在上面开发。

Bug分支

修复bug时，我们会通过创建新的bug分支进行修复，然后合并，最后删除；
当手头工作没有完成时，先把工作现场git stash一下，然后去修复bug，修复后，再git stash pop，回到工作现场。

工作现场处理

$ git stash //把当前工作现场“储藏”起来，等以后恢复现场后继续工作
$ git stash list //查看工作现场存

建分支来修复bug

首先确定要在哪个分支上修复bug，假定需要在master分支上修复，就从master创建临时分支：
$ git checkout master
$ git checkout -b issue-101

处理bug

修复完成后

切换到master分支，并完成合并，最后删除issue-101分支
$ git checkout master
$ git merge –no-ff -m “merged bug fix 101” issue-101
$ git branch -d issue-101

回到dev分支原来工作现场

$ git checkout dev //切换分支
$ git status //查看当前状态
$ git stash list //查看工作现场
$ git stash pop //恢复工作现场的同时把stash内容也删了

存在多次stash工作现场，恢复的时候，先用git stash list查看，然后恢复指定的stash，用命令:$ git stash apply stash@{0},用git stash apply恢复，但是恢复后，stash内容并不删除，你需要用git stash drop来删除;

Feature分支(新的功能要不断添加)

每添加一个新功能，最好新建一个feature分支，在上面开发，完成后，合并，最后，删除该feature分支。
$ git checkout -b
$ git add
$ git commit -m “添加新功能”
$ git merge –on-ff -m
$ git checkout
$ git branch -d //删除一个分支
$ git branch -D //强行删除一个没有被合并过的分支

多人协作

远程仓库

git remote -v

推送分支

把该分支上的所有本地提交推送到远程库
$ git push origin master
$ git push origin dev

抓取分支

$ git clone git@github.com:username/learngit.git //目录下克隆
$ git branch //只能看到本地的master分支
$ git checkout -b dev origin/dev //创建本地dev分支
$ git push origin dev //时不时地把dev分支push到远程

冲突解决

先用git pull把最新的提交从origin/dev抓下来，然后，在本地合并，解决冲突，再推送：
$ git branch –set-upstream //指定本地dev分支与远程origin/dev分支的链接
$ git pull

因此，多人协作的工作模式通常是这样：

  • 首先，可以试图用git push origin branch-name推送自己的修改；
  • 如果推送失败，则因为远程分支比你的本地更新，需要先用git pull试图合并；
  • 如果合并有冲突，则解决冲突，并在本地提交；
  • 没有冲突或者解决掉冲突后，再用git push origin branch-name推送就能成功！
    如果git pull提示“no tracking information”，则说明本地分支和远程分支的链接关系没有创建，用命令git branch --set-upstream branch-name origin/branch-name。

总结：

  • 查看远程库信息，使用git remote -v；
  • 本地新建的分支如果不推送到远程，对其他人就是不可见的；
  • 从本地推送分支，使用git push origin branch-name，如果推送失败，先用git - pull抓取远程的新提交；
  • 在本地创建和远程分支对应的分支，使用git checkout -b branch-name origin/- branch-name，本地和远程分支的名称最好一致；
  • 建立本地分支和远程分支的关联，使用git branch –set-upstream branch-name origin/- branch-name；
  • 从远程抓取分支，使用git pull，如果有冲突，要先处理冲突。

搭建Git服务器

搭建Git服务器需要准备一台运行Linux的机器，强烈推荐用Ubuntu或Debian

安装git

$ sudo apt-get install git

创建一个git用户

$ sudo adduser git

创建证书登录

收集所有需要登录的用户的公钥，就是他们自己的id_rsa.pub文件，把所有公钥导入到/home/git/.ssh/authorized_keys文件里，一行一个。

初始化Git仓库

先选定一个目录作为Git仓库，假定是/srv/sample.git，在/srv目录下输入命令
$ sudo git init –bare sample.git //创建一个裸仓库
$ sudo chown -R git:git sample.git //不让用户直接登录到服务器上去改工作区

禁用shell登录

编辑/etc/passwd文件

克隆远程仓库

$ git clone git@server:/srv/sample.git
